# Deployment guide — publishing the Metrics Encyclopedia

> Three deployment paths. Pick whichever matches your access and audience.

[← Home](./index.md)

---

## Path A — Read locally (zero setup)

GitHub renders Markdown automatically. Click any link from [`index.md`](./index.md) and you're done.

**Limit.** The interactive dashboard at [`dashboard/index.html`](./dashboard/index.html) is HTML — GitHub will show its source rather than render it. For the dashboard, use Path B or C.

✅ Works immediately
❌ Dashboard is source-only

---

## Path B — Open the dashboard locally

```bash
# clone (any branch with this folder)
git clone https://github.com/<user>/torchmetrics.git
cd torchmetrics

# open the dashboard in your default browser
start metrics-encyclopedia\dashboard\index.html      # Windows (PowerShell)
open  metrics-encyclopedia/dashboard/index.html      # macOS
xdg-open metrics-encyclopedia/dashboard/index.html   # Linux
```

The dashboard is a single HTML file plus `data.js`. No server, no build step.

✅ Full interactive experience locally
❌ Requires a clone

## Path C — Publish via GitHub Pages (recommended for sharing)

Two sub-paths depending on whether the repo already has a Pages site (the sibling [`torchmetrics-deep-dive/`](../torchmetrics-deep-dive/) is also a Pages site, so coexistence matters).

### C1. Single-folder Pages (this folder is the only site)

1. Push the branch containing `metrics-encyclopedia/`.
2. Repository **Settings → Pages**.
3. Source: **Deploy from a branch**.
4. Branch: your branch · **Folder: `/metrics-encyclopedia`**.
5. Save. The site builds in 1-2 minutes at `https://<user>.github.io/torchmetrics/`.

✅ Public URL · ✅ Dashboard renders · ⏳ ~2 min on first deploy

### C2. Coexist with `torchmetrics-deep-dive`

GitHub Pages serves from **one** folder. To host both:

**Option 1 — Pick a primary, link to the other.**
Set Pages to one folder; the other is read on GitHub directly. Cross-link the indexes from each `index.md`.

**Option 2 — Promote one folder to the repository root via a wrapper Pages site.**
Create a top-level `_config.yml` and `index.md` that links to both subfolders. Set Pages source to `/` (root). Both sites are served as subpaths.

Layout for option 2 (already partially in this repo):
```
/
├── _config.yml         ← root Jekyll config
├── index.md            ← top-level landing, links to both sites
├── torchmetrics-deep-dive/
└── metrics-encyclopedia/
```

**Option 3 — GitHub Actions workflow.**
Use a custom Pages workflow that bundles both into a single artifact. See `.github/workflows/pages.yml` for an example pattern (the existing `torchmetrics-deep-dive` already has this setup; extending it to include `metrics-encyclopedia` is a 5-line change).

## Path D — Use a different hosting target

Both folders are static-only (Markdown + HTML + JS). Any static host works:

- **Netlify** — drag-and-drop the folder, or connect the repo and set publish dir to `metrics-encyclopedia/`.
- **Vercel** — same pattern.
- **Cloudflare Pages** — same.
- **AWS S3 + CloudFront** — sync the folder to a bucket, point CF at it.
- **Internal corporate static-content host** — copy the folder.

For each: the entry point is `index.html` (auto-generated by Jekyll from `index.md` if Jekyll runs) or `index.md` (rendered by the host's Markdown engine).

## Verifying a deploy

After deploy, check these URLs work:

```
<site-root>/                        ← landing
<site-root>/classification/         ← family page
<site-root>/classification/interview-deep-dive
<site-root>/scenarios/              ← cross-family
<site-root>/dashboard/              ← interactive dashboard
```

If `<site-root>/dashboard/` shows source instead of rendering, your host is mistreating `.html`. For Jekyll, ensure `dashboard/` has `layout: null` in front-matter or set `defaults` in `_config.yml` (already done).

## Local-preview the Jekyll build

If you want to preview the rendered HTML before pushing:

```bash
cd metrics-encyclopedia
gem install bundler jekyll
bundle init
bundle add jekyll
bundle exec jekyll serve
```

Visit `http://localhost:4000/`.

## Updating content

The site is plain Markdown + one HTML file:

| To change | Edit |
|---|---|
| Family page content | `<family>/index.md` |
| Interview Q&A on a family | `<family>/interview-deep-dive.md` |
| Cross-family scenarios | `scenarios/index.md` |
| Dashboard data | `dashboard/data.js` |
| Dashboard look | `dashboard/index.html` (CSS at top) |
| Site nav | `_config.yml` `navigation:` block |

Push and Pages rebuilds automatically.

## Troubleshooting

**"Pages built but a page returns 404."**
Likely a relative-link issue. Jekyll's `jekyll-relative-links` plugin (already in `_config.yml`) usually handles `./family/index.md`-style links, but trailing-slash variants can fail. Use the full `.md` path in markdown, not the directory name.

**"Dashboard shows raw HTML."**
Front-matter conflict. The `defaults` rule in `_config.yml` excludes `dashboard/` from the Jekyll layout. Verify that block exists; if you removed it, add it back.

**"Markdown links broken on GitHub but work after deploy."**
GitHub renders `./classification/index.md` as a link to the file (raw view); Jekyll renders it as a link to the *page*. To work in both: use relative paths and rely on `jekyll-relative-links`.

**"Search box in the dashboard doesn't find a metric."**
Check `dashboard/data.js` — the metric must be in a family's `metrics` array. The full-text search reads `name + what + when + scenario + notWhen + pitfall`.

## Pre-PR checklist

Before pushing changes that affect the site:

- [ ] All new pages have `---\ntitle: …\n---` front-matter.
- [ ] All cross-page links use relative paths (`../family/index.md`).
- [ ] New family pages added to `_config.yml` `navigation:` and to `index.md` table.
- [ ] New metrics also added to `dashboard/data.js`.
- [ ] Local preview at `dashboard/index.html` opens cleanly with the new entry.
- [ ] No emojis added unless requested.
- [ ] Trailing newline on every `.md` file.
